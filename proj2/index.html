Overview

In this project, we learned how to implement 2D and 3D Bézier curves using De Casteljau’s Algorithm. It was intriguing to learn about and implement smooth curves using only a set number of points. This was an area of computer and vector graphics that we knew about but had very little idea how it worked, so it was interesting to implement it as part of the project.

In the second part of the project, we began working with 3D meshes, implementing Blinn-Phong shading using face normals, as well as edge flips, edge splits, and ultimately upscaling of the entire mesh using loop subdivision. It was really unique to work with representations of data-structures as part of meshes themselves, as this was something we didn’t have any experience with. Half-edges are a slightly confusing data-structure but with practice we started to get the hang of accessing the various elements as part of a mesh.

Task 2

De Casteljau algorithm extends to Bézier surfaces because given 4x4 control points and parameters (u, v), we can utilize the algorithm across all row control points according to u. This outputs Bézier curves according to the u parameter. Running these curves in the algorithm according to the v parameter, we will obtain a final point along this curve to return. This image illustrates this concept:

We implemented it by calling evaluate1D() on all control points according to u into some list uVectors. evaluate1D() works by recursively calling a lerp function similar to task 1 until a singular point is yielded from them. Then, we call evaluate1D() on the uVectors to yield the singular final point, creating a Bezier curve along a surface according to interpolation values u and v.

Task 3

Task 3 was fairly simple. What we did to calculate the vector normals is to create an empty sum vector, then iterate over neighboring faces by accessing an initial half-edge, computing the face normal using its built-in function and adding it to the sum vector, then accessing the next relevant half-edge and face by accessing the current half-edge’s twin()→next(). We repeat until we reach the initial half-edge (meaning we’ve traveled to all neighboring faces), then normalize the sum vector by dividing it by its length, returning that vector.

Task 4

In order to implement edge flipping, we followed the recommendations from the spec to create a crude diagram of all elements of the two triangles. We then find and assign references to every single element in these two triangles in order to have full coverage over every pointer we needed to keep track of. We then reassign element half-edges, making sure that all elements besides half-edges are pointed to half-edges associated with the new flipped triangles. We then use the setNeighbors function of the half-edges to assign their associated pointers.

We were fortunate to have this work almost instantly, and we didn’t run into any issues with unassigned or incorrectly set pointers. We were initially confused that some edges would “disappear” after flipping. This makes conceptual sense after thinking through it, as some edges don’t have a defined flipped version (or rather they do, but it becomes extremely hard to see them).

Task 6

In order to implement Task 6, we followed each step provided in the comments of the skeleton thoroughly. First, we iterated over all vertices in the mesh, computing a new position according to spec, and place it in each vertex’s newPosition. We then do the same thing with edges, computing a new position (using a different equation) for the midpoint vertex created when we eventually split edges.

We then split all original edges in the mesh, creating new vertices which we update the position of from the edge’s newPosition. This also creates many new faces and edges across the mesh as described in Task 5.

We then flip new edges belonging to an old and new vertex. We ran into some trouble here, the way we were keeping track of if edges are new or not wasn’t comprehensive. We ended up flipping edges that were split in two, but not necessarily “new,” as the edges were along a previous edge. We solved this by making sure we were only flipping completely new edges. What we did to find this problem was to update the data viewer on the right side of the screen when clicking on an element. We displayed the isNew value when clicking on a vertex to debug which edges should be split.

Finally, we update the position of original vertices calculated all the way back in the initial step. Iterating over all vertices and edges to make sure their isNew value is set to false to allow subsequent subdivisions.
